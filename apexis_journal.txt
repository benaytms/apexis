SQLite stores the entire database in a single file on disk — apod_words.db. 
This works perfectly on PC because that file just sits there permanently.
Cloud servers like Railway work differently. Every time the app redeploys (which happens automatically when it's pushed to GitHub), the server creates a fresh container — like a brand new computer that only has your code on it, nothing else. Any files the app created during runtime, including the SQLite database, get wiped.


deploy → script runs → database created → redeploy → database wiped → back to zero


PostgreSQL is a separate service that runs independently from the app. It's not a file on the same server — it's its own dedicated database server that persists data regardless of what happens to the app. Redeployments, crashes, restarts — the data stays untouched because it lives somewhere completely separate.

deploy → app connects to PostgreSQL → data is always there
redeploy → app reconnects to PostgreSQL → data is still there
